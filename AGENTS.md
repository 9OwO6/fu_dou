# Happy Beans（福豆）项目协作规则

## 1. 语言

- 永远使用中文与用户沟通。
- 代码标识符、命令、第三方产品名可以保留英文。
- 面向顾客的第一版网站文案使用中文；代码结构必须预留英文双语能力。

## 2. 开始任何任务前必须阅读

按以下顺序完整阅读，不要只读摘要或局部：

1. 根目录 `AGENTS.md`。
2. 根目录 `HAPPY_BEANS完整开发计划.md`。
3. `docs/PROJECT_STATUS.md`（如果存在）。
4. 当前 Phase 直接相关的 `docs/` 文档。
5. 当前准备修改的现有代码、测试、迁移和配置。

如果不同文档存在冲突，优先级如下：

1. 用户在当前任务中的明确要求。
2. 根目录 `AGENTS.md`。
3. `HAPPY_BEANS完整开发计划.md` 中已经确认的产品边界。
4. `docs/PROJECT_STATUS.md` 中记录的已完成状态和交接信息。
5. 其他调研、设计和实现文档。

`福豆网站开发.md` 是早期调研资料，不是当前执行范围的最终依据。

## 3. 项目目标

Happy Beans（福豆）是一个可爱家居用品、摆件、餐具、杯具、地毯、玩偶和礼物类商品网站。

第一版提供：

- 中文商品展示网站。
- 分类、搜索、新品、推荐和特价展示。
- 任意商品规格组合。
- 游客购物车。
- 无在线支付的订单请求。
- 管理员商品与首页运营后台。
- 店主后台订单列表和邮件通知。
- 自取和小范围本地配送。

第一版不是完整电商平台，不得把项目自行扩展成 Shopify 替代品。

## 4. 不可改变的 MVP 边界

- 第一版中文，架构从开始预留英文双语。
- 顾客不注册、不登录。
- 顾客以游客身份浏览、使用购物车并提交订单请求。
- 第一版不接 Stripe、PayPal 或其他在线支付。
- 展示币种仅为 CAD，不展示 RMB，不做汇率换算。
- 履约方式仅为自取和小范围本地配送。
- 商品支持颜色、尺寸、款式等任意规格组合。
- 每个具体规格组合拥有独立 SKU、CAD 价格、原价/特价、库存和启用状态。
- 后台支持商品、规格、分类、图片、价格、库存、上下架、新品、推荐和特价管理。
- 管理员可以维护受控首页模块的内容、显隐和顺序。
- 不做自由拖拽页面编辑器，不允许管理员注入任意 HTML 或 JavaScript。
- 不做“特供”功能。

以下功能除非用户另行明确启动后续 Phase，否则不得加入 MVP：

- 在线支付。
- 顾客账号、收藏和订单历史。
- 多币种。
- 自动税费和复杂运费计算。
- 优惠码、积分、会员等级。
- 商品评论与评分。
- 多仓库和采购系统。
- 自动物流标签与追踪。
- Instagram 自动同步。
- 正式英文内容上线。

## 5. 技术边界

- 使用 Next.js App Router，不创建 Pages Router。
- 使用 React、TypeScript 和 Tailwind CSS。
- UI 组件可按需使用 shadcn/ui，不得无目的批量引入组件或依赖。
- 使用 Supabase Postgres、Auth、Storage 和 RLS。
- Supabase Auth 第一版只服务管理员，不开放公众注册。
- 使用 Vercel Preview 和 Production 部署。
- UI 路由从一开始预留 locale；第一版启用 `zh`，未来扩展 `en`。
- UI 系统文案使用 locale 字典，不将大量中文散落硬编码到组件中。
- 商品、分类、集合和首页内容使用可扩展的 translation 数据模型。
- 技术版本必须在实际初始化或升级时根据当前官方文档核对并锁定，不盲目复制早期调研里的版本号。

## 6. Phase 范围控制

- 只实施用户当前明确指定的 Phase 或子阶段。
- 不提前实现后续 Phase。
- 不为了“顺便完善”扩大数据模型、路由、依赖或产品范围。
- 如果当前请求没有写 Phase，但目标明确且能对应开发计划中的某一阶段，应先在工作说明中指出对应 Phase，再按该范围执行。
- 如果一个决定会实质改变产品边界、数据模型或后续路线，停止扩展并向用户说明，不自行决定。
- 发现后续需求时，将其记录到 `docs/PROJECT_STATUS.md` 的“有意延后内容”或“已知问题”，不要提前实现。
- 讨论、评审或规划任务不授权修改业务代码。

## 7. 修改前检查

- 先检查目录结构、现有实现、测试、迁移和配置。
- 先检查 Git/工作区状态；现有修改可能属于用户或其他任务。
- 不覆盖、不回滚、不格式化掉与当前任务无关的修改。
- 优先复用项目已有模式、组件、工具和命名。
- 不做与当前 Phase 无关的大型重构。
- 在 Windows 上处理包含特殊字符的路径时优先使用 `-LiteralPath`。
- 搜索文件和内容优先使用 `rg` 或 `rg --files`。

## 8. 数据库和 Supabase 安全

- 所有暴露 schema 中的表必须启用 RLS，并为实际访问模型编写策略。
- RLS 上线前必须有匿名、非管理员和管理员三类权限验证。
- 数据库结构变更必须使用正式 Supabase migration 工作流。
- 创建迁移文件时使用 Supabase CLI 提供的命令，不伪造迁移文件名。
- 不使用 `service_role` 或 secret key 解决本应由 RLS 处理的权限问题。
- `service_role`/secret key 只能存在于服务器端，绝不能进入浏览器 bundle。
- 不用可由用户修改的 `user_metadata` 决定管理员身份。
- 管理页面保护不能代替 API、Server Action 或数据库层的权限复核。
- 订单请求必须由服务器重新读取商品、规格、价格、特价和库存。
- 不信任 localStorage、URL、隐藏字段或浏览器传来的金额。
- 订单项目保存商品、规格、SKU、价格和图片快照。
- UPDATE 策略必须正确处理 SELECT、USING 和 WITH CHECK。
- Storage 上传必须验证管理员身份、MIME、扩展名、大小和目标路径。
- 数据库、Storage 和邮件操作必须考虑部分失败，不能因邮件失败丢失已经保存的订单请求。
- 修改 RLS、函数、视图、Storage 或 Auth 后必须运行相应安全检查和测试。

## 9. Secret、隐私与日志

- 不读取、打印、复制、提交或在最终答复中暴露 secret。
- 不修改或提交真实 `.env.local`。
- secret 变量不得使用 `NEXT_PUBLIC_` 前缀。
- 仓库只提交不含真实值的 `.env.example`。
- Development、Preview、Production 使用独立配置。
- 顾客订单请求只收集完成联系和履约所需的最少个人信息。
- 第一版不强制收集完整街道地址；店主确认本地配送后再收集必要信息。
- 日志不得记录完整顾客备注、敏感联系方式、密码、token 或邮件服务密钥。
- 测试和截图不得使用真实顾客数据。

## 10. 购物车与订单请求

- 购物车属于游客本地状态，可使用带 schema 版本的 localStorage。
- localStorage 解析失败时必须安全恢复，不能让页面崩溃。
- 不同规格组合必须作为不同购物车项目。
- 加入购物车前必须选择完整且有效的规格组合。
- 售罄、禁用或下架规格不能加入购物车。
- 提交请求时再次进行服务端校验。
- 库存第一版在管理员确认请求后人工扣减，不因顾客提交请求自动永久占用。
- 所有顾客界面和邮件都必须明确说明：提交的是订单请求，尚未付款，也不代表订单已经最终确认。
- 订单状态限定为开发计划定义的状态，不自行创造支付状态流。

## 11. UI 与品牌规则

- 视觉方向以 Happy Beans 自有品牌为准，不复制参考网站。
- 主要使用浅黄色和浅蓝色；粉色和绿色只作小面积品牌强调。
- 整体气质是温暖、快乐、可爱家居与礼物感，可爱但不幼稚。
- 真实商品照片是页面主角；豆子角色用于 Logo、空状态和少量装饰。
- 正式前端页面必须依据用户已经批准的完整视觉概念和 `docs/DESIGN_SYSTEM.md`。
- 如果完整视觉概念尚未批准，不得自行把临时骨架当成最终设计。
- 不只设计或实现 Hero；必须考虑完整页面、下游状态和移动端。
- 保持商品规格、价格、库存和订单提示优先于装饰。
- 所有交互应具备 hover、focus、disabled、loading、empty、success 和 error 状态。
- 必须满足键盘操作、可见焦点、表单标签、错误关联、图片 alt 和颜色对比度要求。
- 尊重 `prefers-reduced-motion`。

## 12. UI 验证

- UI 任务必须使用真实浏览器验证，不能只凭 build 成功判断完成。
- 至少验证常见桌面、笔记本和手机宽度。
- 检查长标题、缺图、售罄、特价、空列表、错误和加载状态。
- 如果存在批准的视觉概念，必须保存实现截图并与概念直接比较。
- 对照检查文案、结构、字体、颜色、图片裁剪、间距、组件状态和响应式行为。
- 存在明显视觉偏差时继续修正，不把可修复问题留给用户验收。

## 13. 代码质量

- 新代码使用 TypeScript，避免无说明的 `any`。
- 输入在客户端提供体验校验，并在服务器端进行权威校验。
- 组件保持职责清楚；避免巨型页面组件和复制粘贴式逻辑。
- 数据访问、权限、校验、邮件和业务规则放在明确的服务/模块层，不散落在 UI 组件中。
- 只在能减少真实重复时抽象，不做预想式过度工程。
- 新增依赖前说明必要性，优先使用现有依赖和平台能力。
- 固定依赖版本并提交 lockfile。
- 不留下失效按钮、假交互、调试代码、临时账号或无说明占位内容。
- 错误信息对用户清楚，对外不暴露数据库和内部实现细节。

## 14. 测试与验证

- 每个 Phase 运行与风险相称的检查。
- 项目具备相应脚本后，通常至少运行：lint、typecheck、相关测试和 production build。
- 数据库 Phase 必须测试迁移重建、约束和 RLS。
- 核心业务 Phase 必须验证真实用户路径，不只验证单个函数。
- 只修复当前改动引入或当前 Phase 明确覆盖的问题；无关旧问题单独记录。
- 不因测试困难而删除或弱化安全约束。
- 不把“命令成功”表述成“用户流程已验证”，除非确实运行了相应路径。

## 15. 文档与状态更新

- 每个实施型 Phase 结束后更新 `docs/PROJECT_STATUS.md`。
- 只在实际满足验收标准后将 Phase 标记完成。
- 数据模型变化同步更新 `docs/DATA_MODEL.md`。
- 架构决定同步更新 `docs/ARCHITECTURE.md`。
- 已批准视觉规则同步更新 `docs/DESIGN_SYSTEM.md`。
- 新增环境变量同步更新 `.env.example` 和相关说明，不写真实值。
- 面向店主的操作变化同步更新 `docs/ADMIN_GUIDE.md`。
- 不在文档中宣称尚未真实验证的功能已经可用。

## 16. Git、部署与外部操作

- 不执行破坏性 Git 命令，不丢弃用户或其他任务的修改。
- 未经用户明确要求，不提交、不推送、不合并分支。
- 非指定部署任务不发布 Production。
- Preview 与 Production 必须使用正确环境和独立邮件策略。
- 上线前按开发计划执行完整验收和 smoke test。
- Production 发布后必须验证网站、后台、订单请求和邮件通知，并确认回滚路径。

## 17. 完成任务时的最终汇报

最终答复必须用中文，并包含：

- 完成内容。
- 修改文件。
- 新增或变更的路由、组件和功能。
- 数据库、API、Storage 和环境变量变化；没有则明确写“无”。
- 执行的检查及其结果。
- 真实浏览器或人工验证结果。
- 用户可以执行的人工验收步骤。
- 未完成项、风险和有意延后内容。
- 当前 Phase 是否真正完成。
- 下一 Phase 是否具备启动条件。
- 是否需要用户执行 Git commit、push、外部账号配置或其他操作。

不要只说“已完成”或“构建通过”。结论必须与实际验证证据一致。

## 18. 新任务最简使用方式

根目录 `AGENTS.md` 会提供每个任务都需要的永久上下文，因此用户不需要再复制通用长提示词。

新任务通常只需要这样说：

```text
请执行 Phase 3，严格按照根目录 AGENTS.md 和 HAPPY_BEANS完整开发计划.md。
```

如果只执行子阶段：

```text
请执行 Phase 5B，不要提前实施 Phase 5C。
```

任务必须自行读取本文件、主开发计划、项目状态和相关文档，不能要求用户重新粘贴已有规则。
