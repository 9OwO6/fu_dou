## 1. 文档用途

本文件用于指导 Happy Beans 网站从空仓库走到可交付 MVP。后续每个开发任务都应先阅读本文件，只实施被指定的 Phase，不提前扩展后续功能。

原文件《福豆网站开发.md》保留为早期调研和技术分析资料；当两份文件出现范围差异时，以本文件中已经确认的产品边界为准。

### 1.1 多任务协作规则

1. 一个任务原则上只完成一个 Phase，复杂 Phase 可以按文档中的子阶段拆分。
2. 每个任务开始前必须读取：
   - 根目录 `AGENTS.md`
   - 本文件
   - `docs/PROJECT_STATUS.md`（在 Phase 1 创建）
   - 当前 Phase 直接依赖的设计或架构文档
3. 开始修改前检查仓库状态，不覆盖其他任务或用户已有的改动。
4. 不跨 Phase 顺手添加功能；发现后续需求时记录到项目状态文档，不提前实现。
5. 每个 Phase 结束必须：
   - 运行该阶段要求的检查
   - 更新 `docs/PROJECT_STATUS.md`
   - 列出改动文件、数据库变更、环境变量和遗留问题
   - 提供人工验收步骤
6. 未通过验收的 Phase 不得标记完成，也不应启动依赖它的下一阶段。

## 2. 项目愿景

Happy Beans 是一家以温哥华和 BC 省顾客为主要服务对象、同时面向加拿大消费者的可爱家居与礼物小店。商品包括杯具、餐具、家居摆件、地毯、玩偶和其他可爱生活用品。

网站第一版的目标不是复制完整 Shopify，也不是立即建立在线支付系统，而是提供一个具有品牌感、方便浏览、可以选择商品规格并提交订单请求的可靠网站，同时让店主可以独立完成日常商品和首页运营。

### 2.1 第一版价值

- 顾客可以清楚浏览商品、规格、价格和库存。
- 顾客无需注册即可使用购物车并提交订单请求。
- 店主可以在后台快速上新、调整库存、设置特价和维护首页内容。
- 店主可通过后台列表和邮件及时处理新订单请求。
- 网站在中文首发的同时，从路由、数据和组件层预留英文扩展能力。

### 2.2 参考方向

- 信息结构和品牌经营感主要参考 Wuli Home。
- 商品导购、分类与新品展示可适度参考 Cutie Pocket。
- 不复制参考站的代码、图片、文案、Logo 或独特视觉资产。
- Happy Beans 必须形成自己的浅黄色、浅蓝色品牌系统。

## 3. 已确认的产品决策

| 事项 | 已确认决定 |
|---|---|
| 网站类型 | 可爱家居与礼物商品展示站 + 购物车 + 订单请求 + 管理后台 |
| 第一版支付 | 不接在线支付 |
| 顾客账户 | 第一版不注册、不登录 |
| 管理员 | 必须登录；至少支持一名店主管理员 |
| 订单方式 | 游客提交订单请求，店主后续联系确认 |
| 通知方式 | 后台订单列表 + 邮件通知 |
| 履约方式 | 自取 + 小范围本地配送 |
| 服务区域 | 重点为温哥华/BC，网站面向加拿大公开访问 |
| 展示币种 | CAD |
| RMB | 可线下协商收取，但第一版网站不展示 RMB 价格、不做汇率换算 |
| 第一版语言 | 中文 |
| 后续语言 | 英文，最终支持中英双语 |
| 商品规格 | 支持颜色、尺寸、款式等任意规格组合 |
| 库存 | 每个具体规格独立管理库存 |
| 运营标签 | 新品、推荐、特价；不做“特供” |
| 视觉方向 | Wuli Home 式家居生活与礼物方向，结合 Happy Beans 品牌 |
| 主色 | 浅黄色、浅蓝色 |

### 3.1 当前采用的合理默认值

以下内容暂未由店主最终决定，但为了推进设计，MVP 采用这些默认值；后续可在不破坏架构的情况下调整。

- 顾客提交：姓名、邮箱、联系电话、偏好联系方式、履约方式、城市/区域、邮编、订单备注。
- 第一版不在请求表单中强制采集完整住址；店主确认订单后再收集必要地址，减少个人信息存储。
- “本地配送”是否可用由管理员配置，不自动计算复杂配送费。
- 购物车显示商品小计；运费、税费和最终金额由店主联系确认。
- 库存只在管理员确认请求后人工扣减，避免恶意请求占用库存。
- 新品可按发布时间自动判断，也允许管理员手动推荐。
- 特价由原价、现价及可选活动起止时间表示。
- 管理后台界面第一版使用中文。

## 6. 核心用户流程

### 6.1 顾客浏览与提交订单请求

```text
首页/分类/搜索
→ 商品详情
→ 选择完整规格组合
→ 检查该规格价格和库存
→ 加入购物车
→ 调整数量
→ 前往订单请求
→ 填写联系与履约信息
→ 确认“这不是已付款订单”
→ 提交
→ 生成请求编号并显示成功页
→ 店主后台和邮箱收到通知
```

### 6.2 管理员上新

```text
管理员登录
→ 新建或复制商品
→ 填写中文商品信息
→ 选择分类
→ 创建规格名称和值
→ 生成规格组合
→ 设置各组合 SKU/价格/库存
→ 上传并排序图片
→ 设置新品/推荐/特价
→ 预览
→ 发布
```

### 6.3 管理员处理订单请求

```text
收到邮件
→ 打开后台订单请求
→ 核对商品与当前库存
→ 联系顾客确认自取/配送、税费和最终价格
→ 将状态改为“已确认”
→ 人工扣减库存
→ 完成后标记“已完成”
```

### 6.4 订单请求状态

- `new`：新请求
- `contacted`：已联系
- `confirmed`：已确认
- `preparing`：准备中
- `completed`：已完成
- `cancelled`：已取消

公开成功页只显示本次请求编号和说明，不提供可枚举的订单查询接口。

## 7. 订单请求表单设计

### 7.1 必填字段

- 姓名
- 邮箱
- 偏好联系方式：邮箱或电话
- 履约方式：自取或本地配送
- 所在城市/区域
- 同意订单请求与隐私说明

### 7.2 条件必填字段

- 选择电话联系时：联系电话必填
- 选择本地配送时：邮编必填

### 7.3 可选字段

- 联系电话（偏好邮箱时）
- 微信号或其他补充联系方式
- 期望取货/配送时间
- 订单备注

### 7.4 请求确认文案

提交按钮附近必须清楚表达：

> 提交后不会立即付款，也不代表订单已经确认。Happy Beans 将联系您核对库存、自取或配送安排、税费及最终金额。

### 7.5 反滥用

- 服务端校验所有字段。
- 加入隐藏 honeypot 字段。
- 对同一 IP/邮箱做温和频率限制。
- 后续如真实滥用明显，再加入 Turnstile；MVP 不预先增加验证码摩擦。
- 邮件和后台均不得直接渲染未经转义的顾客输入。

## 8. 管理员可控内容边界

后台的“调整网站布局”定义为受控内容编排，不做自由拖拽建站器。

### 8.1 首页模块

- 公告条
- Hero 主视觉
- 热门分类
- 新品
- 推荐商品
- 特价商品
- 品牌故事
- 自取/本地配送说明
- FAQ 摘要
- 联系 CTA

### 8.2 管理员可执行的操作

- 修改模块标题、简介、图片和按钮文案
- 开启或隐藏模块
- 调整模块顺序
- 选择模块展示的分类或商品
- 设置每个商品区显示数量
- 修改公告条

### 8.3 管理员不能执行的操作

- 写入任意 HTML 或 JavaScript
- 任意修改网站网格和 CSS
- 创建未经过设计的新模块类型
- 绕过既定组件直接上传整页代码

这样可以在保持运营灵活性的同时避免布局损坏和安全风险。

## 9. 视觉与品牌方向

### 9.1 品牌基调

- 关键词：温暖、快乐、轻松、可爱家居、礼物感、小店亲和力。
- 视觉成熟度：可爱但不幼稚，适合成人购买家居用品和礼物。
- 页面应以真实商品照片为主，品牌豆子角色用于 Logo、空状态和少量装饰。
- 不让插画压过商品信息。

### 9.2 暂定颜色令牌

以下为根据当前 Logo 提炼的初步方向，正式值必须在视觉概念阶段通过取色和对比度测试确定。

| 用途 | 暂定颜色 |
|---|---|
| 主黄色 | `#F7E653` |
| 主浅蓝 | `#A9E3EA` |
| 品牌粉 | `#F28FB2` |
| 品牌绿 | `#89C43F` |
| 页面背景 | `#FFFDF8` |
| 主要文字 | `#2F3133` |
| 柔和边框 | `#E9E4D9` |
| 成功/有库存 | `#4F8A4C` |
| 危险/售罄 | `#B85454` |

浅黄色与浅蓝色是主视觉；粉色和绿色仅作为小面积强调色。

### 9.3 组件气质

- 大圆角但不把每个区域都做成卡片。
- 柔和边框和克制阴影。
- 商品卡以图片、标题、价格和状态为核心。
- 按钮有清楚的主次层级。
- 商品规格选择必须比装饰更清晰。
- 商品图使用稳定的统一比例；详情页支持原始比例查看。
- 所有交互具备 hover、focus、disabled、loading 和 error 状态。

### 9.4 Logo 与资产要求

正式视觉实施前需要准备：

- 透明背景 PNG Logo
- SVG 或其他矢量 Logo
- 横版 Logo
- 豆子角色图标版
- favicon/应用图标版
- 品牌名称最终拼写确认：当前按 `Happy Beans` 处理
- Logo 与主要商品图片的使用授权确认

### 9.5 视觉批准门槛

前端正式编码前必须先生成并批准完整视觉概念，包括：

- 桌面首页各主要区段
- 移动端首页
- 商品列表与商品卡
- 商品详情及规格选择
- 购物车与订单请求
- 管理后台商品编辑页

只有 Hero 图不算完整视觉方案。批准后的概念应转化为设计令牌和组件规范，并在实施阶段进行截图对照验证。

## 10. 技术架构

### 10.1 建议技术栈

- Next.js App Router
- React + TypeScript
- Tailwind CSS
- shadcn/ui（按需引入，不批量安装）
- Supabase Postgres
- Supabase Auth（仅管理员）
- Supabase Storage（商品图和内容图）
- Supabase RLS
- Vercel Preview 与 Production
- 事务邮件服务：优先评估 Resend，最终在订单通知阶段确认
- 测试：Vitest 或项目选定的单元测试工具 + Playwright E2E

具体版本在 Phase 1 实施时根据官方文档和当前稳定版本锁定并提交 lockfile，不直接照抄早期调研文档中的旧版本尾号。

### 10.2 架构原则

- 公开商品读取优先使用 Server Components。
- 管理写操作仅在服务器端执行并验证管理员身份。
- 前端永远不暴露 Supabase secret/service role key。
- 浏览器只可获得 Supabase URL 和 publishable/anon key。
- 所有公开 schema 表启用 RLS。
- 订单请求必须由服务端重新读取商品和规格价格，不能相信购物车中的客户端价格。
- 订单项目保存标题、规格和价格快照，防止商品后续修改影响历史请求。
- 页面内容采用受控模块，不允许管理员写任意脚本。
- 图片文件路径由系统生成，不直接信任用户文件名。

## 11. 国际化策略

第一版只显示中文，但必须避免把所有中文硬编码到几十个组件中。

### 11.1 UI 文案

- 使用 locale 字典管理导航、按钮、表单、错误和系统文案。
- 初始 `zh` 字典完整。
- 初始 `en` 可以只保留结构或在英文 Phase 再补内容。
- 路由从第一天采用 `[locale]`，避免后期整体迁移。

### 11.2 商品与内容

- 商品、分类、集合和首页内容采用基础表 + translation 表。
- `zh` 是第一版必填内容。
- `en` 允许为空；英文上线前后台补齐翻译。
- SKU、库存、价格和图片属于语言无关数据，不复制。

### 11.3 SEO

- 每个 locale 独立 metadata。
- 英文未完成时不公开英文页面，也不进入 sitemap。
- 英文正式上线后添加 hreflang/canonical。

## 12. 数据模型

以下是逻辑模型。具体 SQL、约束和索引在数据库 Phase 中完成并单独评审。

### 12.1 管理员

#### `profiles`

- `id`：关联 `auth.users.id`
- `display_name`
- `role`：第一版只允许 `admin`
- `created_at`
- `updated_at`

管理员角色不能由浏览器自行修改。首名管理员通过受控 seed/SQL 或 Supabase 后台建立。

### 12.2 分类

#### `categories`

- `id`
- `slug`
- `sort_order`
- `is_visible`
- `created_at`
- `updated_at`

#### `category_translations`

- `category_id`
- `locale`
- `name`
- `description`

唯一约束：`category_id + locale`。

### 12.3 商品

#### `products`

- `id`
- `slug`
- `status`：`draft | published | archived`
- `default_currency`：第一版固定 `CAD`
- `published_at`
- `is_featured`
- `created_by`

## 13. 权限模型与 RLS

| 数据 | 游客 | 管理员 |
|---|---|---|
| 已发布商品/分类 | 只读 | 全部读写 |
| 草稿/归档商品 | 不可见 | 可读写 |
| 商品规格和图片 | 仅已发布商品的有效数据可读 | 全部读写 |
| 首页公开内容 | 只读启用内容 | 全部读写 |
| 订单请求 | 仅可通过受控服务端接口提交 | 可查看和更新 |
| 订单请求列表 | 不可读取 | 可读取 |
| 管理员资料 | 不可读取 | 仅必要范围 |
| Storage 商品图片 | 公开商品可读；不可写 | 可上传、替换和删除 |

安全要求：

- 所有公开 schema 表启用 RLS。
- 不用可由用户修改的 metadata 判断管理员权限。
- 管理员判断统一封装并审计，不在每张表里写互相矛盾的逻辑。
- `service_role` 或 secret key 仅存在于服务器端环境变量。
- 订单提交 API 重新读取数据库中的有效规格、库存和价格。
- UPDATE 策略同时考虑 SELECT、USING 和 WITH CHECK。
- 所有管理 API 除页面保护外，还必须在操作层再次鉴权。
- 上传同时校验 MIME、扩展名、大小和管理员身份。

## 14. 图片与内容策略

### 14.1 商品图要求

- 支持 JPEG、PNG、WebP；实施时根据当前浏览器和工具链决定是否转换为 AVIF/WebP。
- 建议单图上传前压缩到合理尺寸和体积。
- 商品列表使用统一主图比例，建议 4:5 或 1:1，在视觉概念阶段决定。
- 详情页保留多图和清楚放大查看。
- 每张图片保存 alt 文案。
- 同一规格可以拥有专属图片。
- 删除商品时不要未经确认直接永久删除仍被订单快照引用的资产。

### 14.2 真实商品图片现状

当前图片适合证明商品和风格，但部分为门店陈列照片、手持照片或供应商图。正式上线前应尽量为主推商品准备：

- 干净背景主图
- 生活场景图
- 尺寸参照图
- 不同规格对应图
- 包装或材质细节图

后台必须允许先用现有照片上线，后续替换主图，不把“摄影全部完成”设为开发阻塞条件。


规则：

- 桌面宽屏。
- 常见笔记本宽度。
- 手机窄屏。
- 首页全部模块。
- 商品长标题、缺图、售罄和特价状态。
- 管理后台长表格和移动端基本可操作性。

### 可访问性

- 可见焦点。
- 表单 label 和错误关联。
- 图片 alt。
- 足够颜色对比度。
- 抽屉和对话框焦点管理。
- 尊重 `prefers-reduced-motion`。

### 安全 QA

- RLS 与管理 API 双重鉴权。
- secret 不进入客户端 bundle、日志或 Git。
- 订单输入转义和校验。
- 上传限制。
- 频率限制。
- 生产错误不暴露数据库细节。

## 18. MVP 验收标准

满足以下全部条件，才可以称为第一版完成：

- 中文公开网站可正常访问。
- 品牌视觉经过用户批准并与实现一致。
- 顾客可浏览真实商品和所有有效规格。
- 每个规格显示正确 CAD 价格和库存。
- 游客购物车刷新后仍保留。
- 顾客可完成自取或本地配送订单请求。
- 服务端不会相信客户端价格和库存。
- 请求成功后后台能看到完整快照。
- 店主收到通知邮件，顾客收到确认邮件。
- 管理员能维护商品、规格、分类、图片、库存、发布状态和特价。
- 管理员能维护受控首页模块。
- 非管理员无法读取后台数据和订单列表。
- 桌面和手机核心路径通过真实浏览器验证。
- lint、typecheck、tests 和 production build 通过。
- Vercel Production 已完成 smoke test，并有明确回滚方法。
- 管理员操作手册和发布检查表已完成。

## 19. 风险与应对

| 风险 | 应对方案 |
|---|---|
| 商品规格复杂度被低估 | 从第一天采用灵活 option/value/variant 模型，并用三类真实商品验证 |
| 店主更新频繁导致后台难用 | 提供复制商品、批量规格编辑、图片排序和清晰状态筛选 |
| “调整布局”不断扩大 | 限定为已批准模块的内容、显隐和排序，不做通用建站器 |
| 订单请求被误认为已付款 | 所有关键步骤明确提示，邮件和后台使用“请求”术语 |
| 垃圾请求 | honeypot、限流、服务端校验；必要时再加 Turnstile |
| 商品图片质量不一致 | 统一主图比例和裁剪；后台允许逐步替换，不阻塞初版 |
| 双语后期重构 | 第一版就建立 locale 路由、字典和 translation 表 |
| Supabase 权限错误 | RLS 测试、服务端复核、advisors 与独立权限验收 |
| 邮件发送失败丢单 | 先落库再发邮件，记录状态并允许重试 |
| 需求逐渐变成完整商城 | 坚守 MVP 非目标，支付和顾客账户另立 Phase |
| 参考站侵权或品牌素材风险 | 只借鉴结构与体验；使用自有品牌、文案、图片和授权商品素材 |

## 24. 当前建议的下一步
