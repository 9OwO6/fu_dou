# Happy Beans（福豆）设计系统

> 状态：Phase 2 已批准基线  
> 批准日期：2026-07-13（America/Vancouver）  
> 适用范围：中文公开网站、购物车与订单请求、中文管理后台  
> 依据：用户提供的 Happy Beans Logo、三张真实商品照片及 2026-07-13 批准的完整视觉概念

## 1. 设计目标与边界

Happy Beans 是面向成人顾客的可爱家居与礼物小店。界面应温暖、轻松、有礼物感，但不能幼儿化，也不能让插画压过商品信息。

设计优先级：

1. 真实商品照片、商品名称、CAD 价格、规格和库存。
2. 清楚的浏览、加入购物车和提交订单请求路径。
3. 自取、本地配送及“尚未付款、尚未最终确认”的准确说明。
4. 品牌气氛和少量豆子角色装饰。

本系统只借鉴家居礼物站的内容组织方式，不复制 Wuli Home 或其他参考站的独特布局、资产、文案和视觉细节。

明确不包含：在线支付、顾客账号、评分评论、优惠码、多币种、自由拖拽建站器和任意 HTML/JavaScript 编辑器。Phase 12 已启用正式英文顾客端，但后台操作界面仍固定中文。

## 3. 品牌原则

### 3.1 关键词

- 温暖、快乐、轻松。
- 可爱家居、日常小物、礼物感。
- 本地小店的亲和力和可信度。
- 可爱但不幼稚，活泼但不喧闹。

### 3.2 Logo

- 权威来源：`assets/brand/happy-beans-logo-primary.jpg`。
- 不重绘、不改字、不换色、不拉伸、不旋转、不添加阴影或描边。
- 当前文件为方形白底 JPEG。公开 Header 优先显示经安全裁切的方形版本，并为文字导航保留空间。
- Header 桌面建议视觉高度 `48–56px`，移动端 `36–44px`；保持原始宽高比。
- Logo 四周至少保留一个英文大写字母 `H` 高度的安全区。
- 在获得透明或横版源文件前，不从当前 JPEG 自动抠图后冒充正式 Logo 版本。
- 豆子角色只出现在 Logo、空状态、订单提示或少量模块装饰中；同一视口通常不超过两处明显角色图。
- 顾客端空状态、错误页和 404 页统一使用权威 Logo 中的双豆品牌头像，不使用单独的“豆”文字或重新绘制的角色图。

## 4. 颜色系统

浅黄与浅蓝是大面积品牌色；粉色和绿色只承担小面积强调或状态提示。语义状态不能仅依靠颜色表达。

|---|---|---|
| `canvas` | `#FFFDF8` | 全站暖白页面背景 |
| `surface` | `#FFFFFF` | 表单、商品卡、后台面板 |

| `brand-yellow` | `#F7E653` | 主行动、核心品牌强调 |
| `brand-yellow-soft` | `#FFF7C2` | Hero、公告、提示底色 |
| `brand-blue` | `#A9E3EA` | 品牌分区、辅助重点 |
| `brand-blue-soft` | `#E8F7F9` | 履约、信息提示、后台背景 |
| `brand-pink` | `#F28FB2` | 特价、新品、小面积装饰 |

使用规则：

- 默认文字使用 `text-strong`，不要使用品牌黄、浅蓝、粉或亮绿作为正文色。
- `brand-yellow` 按钮使用 `text-strong`；不得配白字。
- `brand-blue`、`brand-pink` 和 `brand-green` 主要作为背景、边框、图标或小标签，不承担长段正文。
- 状态文字优先使用 `success`、`warning`、`danger`、`info`，并配图标或明确文字。

## 5. 字体与文字层级

顾客展示界面采用“可爱展示字体 + 清晰功能字体”的双字体系统。甲方于 2026-07-15 选择 Plan B：英文展示标题使用 `Fredoka`，中文展示标题使用 `ZCOOL KuaiLe / 站酷快乐体`。两款字体通过 Next.js `next/font` 在构建时自托管，仅挂载到公开 locale 布局；后台管理界面继续使用既有系统无衬线字体，不受影响。

展示字体只用于 Hero、首页模块、品牌故事、商品列表页、商品详情页、购物车、订单请求、成功/错误页的主要展示标题，以及 Header 品牌名称。商品卡名称、价格、SKU、库存、规格、按钮、导航、表单、政策/履约正文和 Footer 联系信息继续使用清晰无衬线字体。中文展示字体只使用其原生 `400` 字重，不通过浏览器合成粗体。

功能与正文使用清晰、成熟、略圆润的系统中文无衬线栈：

```css
font-family:
  "PingFang SC",
  "Microsoft YaHei",
  "Noto Sans CJK SC",
  "Noto Sans SC",
  system-ui,
  sans-serif;
```

| 层级 | 桌面 | 移动 | 字重/行高 | 用途 |
|---|---:|---:|---|---|
| `display` | `48px` | `34px` | `700 / 1.15` | Hero 主标题 |
| `h1` | `36px` | `30px` | `700 / 1.2` | 页面标题 |
| `h2` | `28px` | `24px` | `700 / 1.25` | 首页模块标题 |
| `h3` | `21px` | `19px` | `650 / 1.35` | 卡片组和表单区标题 |
| `body-lg` | `18px` | `17px` | `400 / 1.65` | Hero 与说明文案 |
| `body` | `16px` | `16px` | `400 / 1.6` | 正文、表单、按钮 |
| `body-sm` | `14px` | `14px` | `400 / 1.5` | 辅助信息 |
| `caption` | `12px` | `12px` | `500 / 1.45` | 标签、metadata；不用于关键说明 |

- 中文正文不使用全大写或过宽字距。
- CAD 金额使用等宽数字特性 `font-variant-numeric: tabular-nums`。
- 商品长标题默认最多两行；列表卡不靠缩小字号解决溢出。
- 正式文案通过 locale 字典或 translation 数据提供，不在组件中散落硬编码。英文和中文使用同一组件与版式，语言切换必须保留当前页面。
- 英文内容不得回退显示中文；缺少完整英文商品、规格或图片替代文字时，使用英文空状态或隐藏相应实体。
- 英文展示标题使用 `Fredoka`，其余英文内容沿用现有 sans 字体栈；按钮和导航不依赖固定中文字符宽度。长英文标题需允许合理换行，Header 品牌名在窄屏安全截断且不得造成横向滚动。

## 6. 布局与响应式

### 6.1 断点和容器

| 名称 | 范围 | 核心规则 |
|---|---|---|
| `mobile` | `< 640px` | 单列；16px 页面边距；底部操作可吸附 |
| `tablet` | `640–1023px` | 2–3 列商品网格；24px 页面边距 |
| `laptop` | `1024–1279px` | 3–4 列；32px 页面边距 |
| `desktop` | `>= 1280px` | 最大内容宽度 `1280px`；4 列为主 |

- 宽屏内容居中，最大宽度 `1280px`；Hero 可使用全宽背景，但文字和商品内容仍对齐容器。
- 页面最小支持宽度 `320px`，不得产生页面级横向滚动。
- Header 桌面高约 `72px`，移动端 `60px`；公告条约 `32–36px`。
- 商品列表：移动 2 列、平板 3 列、桌面 4 列；极窄屏可降为 1 列。
- 后台桌面优先；窄屏表单改为分区折叠或 variant 卡片，不把整页强行缩小。

### 6.2 间距

基础单位为 `4px`：`4, 8, 12, 16, 20, 24, 32, 40, 48, 64, 80, 96`。

- 首页大模块桌面垂直间距 `72–96px`，移动 `48–64px`。
- 卡片内部常用 `16–20px`，表单分组 `24–32px`。
- 可点击控件最小高度 `44px`，主要按钮建议 `48px`。
- 紧密后台表格行高不低于 `48px`。

## 7. 形状、边框与阴影

| Token | 值 | 用途 |
|---|---:|---|
| `radius-sm` | `8px` | 标签、紧密输入 |
| `radius-md` | `12px` | 按钮、输入框、后台面板 |
| `radius-lg` | `18px` | 商品卡、提示面板 |
| `radius-xl` | `28px` | Hero 图片、重点品牌区 |
| `radius-pill` | `999px` | 状态标签、筛选 chip |

- 不把每个首页区段都包成卡片；交替使用开放留白、浅色整块背景和少量面板。
- 默认边框 `1px solid border-subtle`；选中项使用 `2px` 品牌或焦点色，且不能造成布局跳动。
- 阴影只用于浮层和悬浮反馈：
  - 卡片 hover：`0 8px 24px rgba(41, 44, 48, 0.08)`。
  - 抽屉/对话框：`0 18px 48px rgba(41, 44, 48, 0.16)`。
- 后台面板几乎不用阴影，以边框和间距组织密度。

## 8. 图片系统

- 商品列表和推荐卡主图统一 `4:5`，使用 `object-fit: cover`。
- 分类图建议 `1:1`；Hero 桌面建议约 `16:9` 或 `3:2`，移动改为 `4:5` 或上下结构。
- 商品详情主图区域支持原始比例查看，缩略图建议 `1:1`。
- 不用 AI 重绘、自动替换或过度修饰真实商品；允许非破坏性裁切、曝光和白平衡微调，但不得改变商品颜色、纹样或数量。
- 缺图状态使用暖白/浅蓝底、简洁商品轮廓和文字，不伪造商品照片。
- 每张图片必须有准确中文 alt；纯装饰豆子图使用空 alt。
- 售罄商品图片可以轻微降低饱和度，但仍应清楚；必须同时显示“已售罄”文字。

## 9. 核心组件

### 9.1 按钮

- `primary`：`brand-yellow` 背景、`text-strong` 文字；用于每个区域唯一主行动。
- `secondary`：白色或 `brand-blue-soft` 背景、清晰边框。
- `tertiary`：文字按钮，必须有 hover 和可见焦点。
- `danger`：白底危险色边框或危险色实底，仅用于删除等破坏操作。
- loading 保留原尺寸并显示进度；disabled 不能仅降低透明度，需保留可读标签和禁用语义。

### 9.2 表单

- label 永远可见，不用 placeholder 代替 label。
- 输入框默认高 `44–48px`，边框 `border-subtle`，focus 使用 `2px focus` 环。
- 必填用文字或 `*` 并提供统一说明；错误紧邻字段并通过 `aria-describedby` 关联。
- 错误信息说明如何修复，不暴露数据库或内部字段。
- 订单请求表单在提交按钮前重复显示“未付款、未最终确认”说明。

### 9.3 商品卡

固定信息顺序：图片 → 状态标签 → 商品名 → 规格摘要（可选）→ CAD 价格/原价 → 库存状态。

- 卡片整块可进入详情，但内部按钮必须有独立可访问名称。
- 特价同时显示现价、删除线原价和“特价”标签。
- 售罄卡仍可浏览详情，但购买行动禁用并显示原因。
- hover 只轻微上移 `2px` 和增加柔和阴影；reduced motion 下取消位移。

### 9.4 规格选择

- 规格按业务顺序逐组显示，例如“款式”后“颜色”。
- 文字规格使用按钮/chip；颜色必须同时显示文字名称，不使用纯色块作唯一标识。
- 状态必须覆盖：默认、hover、focus、selected、unavailable、sold-out、disabled。
- 不可用组合保留可见但用斜线/图标和文字说明；不要让选项无解释消失。
- 未选完整规格前“加入购物车”禁用，并清楚提示仍需选择的规格。
- 无顾客可选规格的商品不显示空选择器。

### 9.5 状态与反馈

- loading：骨架屏保持最终布局，避免大幅跳动。
- empty：简短原因、下一步行动和少量品牌角色。
- success：说明已完成什么、下一步是什么。
- error：解释失败和恢复方式；保留用户已输入的非敏感内容。
- toast 只用于轻量确认，不承载必须阅读的订单或库存错误。

## 10. 页面规范

### 10.1 桌面首页

默认模块顺序：

1. 公告条：自取/配送或营业提示，单行且可关闭。
2. Header：Logo、商品分类、新品、推荐、特价、品牌故事、搜索、购物车。
3. Hero：品牌短句、主要 CTA、真实主推商品；桌面左右分栏，移动上下堆叠。
4. 热门分类：5–6 个简洁分类入口。
5. 新品上架：四列商品卡。
6. 本周推荐：浅蓝分区，强调使用场景和真实商品。
7. 温柔特价：浅粉小面积分区，清楚显示原价和现价。
8. 品牌故事：短文、真实门店/商品图；不编造店主事实。
9. 自取与本地配送：浅蓝信息区，说明费用与范围待确认。
10. FAQ 摘要：3–5 个关键问题。
11. 联系 CTA：低压力联系入口。
12. Footer：导航、政策、联系方式和地区说明。

管理员可调整受控模块的内容、显隐、顺序和选品，但不能创建任意新模块或写入 HTML/JavaScript。

### 10.2 移动首页

- 保持与桌面相同的信息顺序，不因移动端只保留 Hero。
- 使用紧凑 Header、清楚的菜单和购物车数量；不强制使用底部导航，实施时按可用性选择。
- 商品横滑区必须有清楚的滚动线索，也要支持键盘/辅助技术；关键列表可直接两列展示。
- Hero 文字先于或紧邻主图，不把 CTA 压在不可读的照片区域。

### 10.3 商品列表

- 页面标题和结果数量在筛选之前。
- 搜索、分类、集合、库存/特价筛选和排序使用 URL 状态。
- 桌面筛选可横向或侧栏；移动使用可访问抽屉并显示已选数量。
- 覆盖长标题、缺图、售罄、特价、空结果、加载和错误状态。

### 10.4 商品详情

- 桌面两栏：左侧画廊，右侧购买信息；移动先主图再信息。
- 信息顺序：面包屑、标题、价格、库存、规格、数量、加入购物车、履约说明、详情/材质/尺寸/护理。
- 价格和库存随完整 variant 选择更新。
- 主图与缩略图支持键盘操作、准确 alt 和放大查看。
- 订单请求性质在购买区附近可见，但不压过规格操作。

### 10.5 购物车

- 每项显示商品图、标题、完整规格、单价、数量、行小计和删除操作。
- 商品下架、价格变化或库存不足以行内警告呈现，并阻止无效提交。
- 小计区明确写明：税费、配送费及最终金额由店主确认。
- 主行动文案为“填写订单请求”，不得使用“结账”或“付款”。

### 10.6 订单请求

- 分组：联系信息 → 偏好联系方式 → 履约方式 → 条件字段 → 备注 → 隐私与请求说明 → 订单摘要。
- 选择电话时电话必填；选择本地配送时邮编必填。
- 不在第一步强制收集完整街道地址。
- 提交按钮使用“提交订单请求”。按钮附近必须显示：`提交后不会立即付款，也不代表订单已经确认。我们会联系你核对库存、自取或配送安排、税费及最终金额。`
- 成功状态显示不可轻易枚举的请求编号和下一步，不显示“付款成功”。

### 10.7 后台商品编辑

- 公开品牌语言延续到后台，但色彩和装饰更克制。
- 后台 Shell 使用暖白画布、白色顶栏和带线性图标的固定侧栏；当前导航以浅黄色底和左侧强调线表示，主操作统一使用品牌黄色配深色文字。
- 桌面左侧导航，主区按基本信息、中文内容、分类、图片、规格、variants、价格库存、运营状态组织。
- 商品详情的基础内容、SEO、分类、图片、规格/价格/库存和运营状态使用原生 `details/summary` 折叠面板；收起时只保留标题、说明和箭头，展开后才显示编辑控件。键盘可操作，错误所在面板应自动展开。
- 分类管理表单固定按业务顺序分组：中英文名称一行、中英文描述一行、网址标识与排序值一行；现有分类默认使用紧凑摘要行，点击“编辑”或整行后展开完整表单。
- 保存、预览和发布状态保持可见；离开前处理未保存更改。
- 图片区支持上传、预览、排序、封面、删除和规格关联。
- variant 矩阵固定显示规格组合、SKU、CAD 价格、原价、库存、启用状态；错误必须定位到具体行和字段。
- 窄屏使用分区折叠和 variant 卡片，不能依赖缩小整张桌面表格。
- 禁止任意 HTML/JavaScript 编辑和自由拖拽页面搭建。

## 11. 交互与动效

- 常规反馈 `120–180ms`，抽屉/对话框 `180–240ms`。
- 缓动建议 `cubic-bezier(0.2, 0.8, 0.2, 1)`。
- 动效只用于状态变化、层级和反馈，不做持续漂浮或大面积视差。
- `prefers-reduced-motion: reduce` 时取消非必要位移、缩放和自动滚动，保留即时状态变化。
- hover 不能是获取信息的唯一方式；触屏和键盘都能完成同一操作。

## 12. 可访问性验收

- 语义化 heading 顺序，页面只有一个主要 `h1`。
- 所有交互可通过键盘完成，焦点清楚且顺序符合视觉流。
- 抽屉/对话框进入时聚焦、关闭后恢复焦点并正确限制焦点范围。
- 表单 label、必填、帮助和错误信息通过程序关系关联。
- 图标按钮有可访问名称；状态不只依靠颜色或图标。
- 触控目标至少 `44 × 44px`。
- 图片 alt 准确描述商品与视角；装饰图 alt 为空。
- 桌面、笔记本和手机宽度均无页面级横向溢出。

## 13. 实施与视觉 QA 门槛

Phase 6、7、8 和相关后台 UI 实现时必须：

1. 以本文件和三张批准概念为基线，不把当前 Phase 1 临时骨架延伸成正式设计。
2. 使用真实浏览器验证常见桌面、笔记本和手机宽度。
3. 保存实现截图，并按模块对照文案、层级、字体、颜色、图片裁剪、间距、组件状态和响应式。
4. 覆盖长标题、缺图、售罄、特价、空列表、加载、错误、表单校验和禁用状态。
5. 对任何明显偏差继续修正；若需改变本文件的关键视觉或信息架构，先获得用户批准并同步更新本文档。

## 15. 顾客端真实信息与图片交互补充（2026-07-15）

- 品牌故事使用店主确认的中英文正式文案；正文按空行渲染为独立段落，避免长文挤成单一文字块。
- Hero 两张真实商品图采用有机卵形遮罩。图片首次进入视口时从下方 30px 淡入，悬停时仅内部图片放大至 `1.05`；所有非必要位移与缩放继续受 `prefers-reduced-motion` 约束。
- 配送区保留店主提供的竖版配送海报完整信息，不使用会截掉金额、范围或客服说明的横向强裁切。桌面以正文/面板和竖版海报双栏呈现，移动端上下堆叠。
- Footer 顶部增加真实门店照片、完整地址、Google Maps 外链、微信二维码与 Instagram 二维码；二维码保持原始比例与足够扫描尺寸，移动端并排显示且不得造成页面级横向滚动。
- 商品详情主图点击后在当前页面打开模态灯箱，不再新标签页打开图片。灯箱支持打开动画、Escape/关闭按钮、背景点击、方向键、前后按钮和多图缩略图；单图时隐藏无意义的切换操作。

## 16. 快速新品墙视觉规则（2026-07-18）

- 新品墙以真实商品照片为主角，使用浅黄到暖白的大面积柔和背景、少量粉色/绿色有机装饰和品牌展示字体；不得把页面做成密集库存表或传统电商分类页。
- 标签使用横向胶囊筛选，手机端允许组件内部横向滚动但不得造成页面级横向溢出。商品卡采用紧凑图片墙节奏，标题和价格可缺省，商品短编号与“请私信确认”始终可见。
- 已售完状态同时使用清晰文字和倾斜印章，不只依靠颜色；归档状态不在公开页出现。多图数量角标、短编号和咨询状态不得被图片动画遮挡。
- 灯箱使用原生 `dialog`，支持关闭按钮、Escape、方向键、前后按钮、缩略图和复制商品编号；打开后焦点进入对话框，关闭后回到触发卡片。
- 卡片进入只允许短距离、单次的淡入/上移，悬停只做轻微抬升；`prefers-reduced-motion` 下移除非必要位移、缩放和延时。
- 快速上新后台保持现有专业管理视觉，但上传入口优先手机操作：大触控区、默认一图一商品、合并/拆分和批量标签在发布前完成，选填字段不得阻止无文字快速发布。
